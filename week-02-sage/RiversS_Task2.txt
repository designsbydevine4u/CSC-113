Here’s my "Kevin from IT" guide to getting the most out of our collaboration this semester:

1. The "Golden Rule": Use Me to Learn, Not Just to Finish
If you just ask me to "write a Python script that sorts a list," I’ll do it, but you won't learn why the code works. When the midterm comes around and I’m not there, that’s when the "brain fog" hits.
* Better prompt: "I need to sort a list in Python. Can you explain the logic behind a 'Bubble Sort' and show me a simple example?"
* The "Explain-Back" Rule: After I give you an answer, try to explain it back to me in your own words. If you can’t, we haven't finished the lesson yet!
2. The "Context is King" Strategy
I don't know what's happening on your screen unless you tell me. The more context you give, the better my "IT brain" can help.
* Specify the Language: "I'm working in Python 3.12..."
* Share the Goal: "...and I'm trying to make a simple calculator for my CSC-113 homework."
* Describe the Error: Don't just say "it's broken." Copy and paste the exact error message and the specific line of code it’s complaining about.
3. Debugging Like a Pro
When your code crashes (and it will—that's just part of the process!), try this three-step prompt:
1. "Here is my code: [Paste Code]"
2. "Here is the error I got: [Paste Error]"
3. "Can you explain what this error means and give me a hint on how to fix it?" (Asking for a "hint" first instead of the full fix is a huge level-up for your brain!)
4. Beware of "Hallucinations"
Sometimes, I get a little too confident and might suggest a library or a function that doesn't actually exist. We call these "hallucinations."
* Always Verify: If I suggest something that looks weird, ask me: "Is that part of the standard Python library, or do I need to install something?"
* Trust the Docs: If your textbook says one thing and I say another, trust your textbook. ---

The "Clean Workbench" Structure
For a CSC-113 project, I recommend this layout. It looks professional and keeps your exploratory "messy" code separate from your "final" work.
Folder/FileWhat goes inside?src/The Engine. Your final, clean Python or Sage scripts.notebooks/The Lab Notes. Jupyter notebooks (.ipynb) where you experiment and visualize data.data/The Ingredients. Any CSVs or text files your program needs to read.docs/The Manual. Your project prompt, screenshots, or PDF reports.README.mdThe Front Door. A brief explanation of what the project is and how to run it..gitignoreThe Trash Filter. A special file that tells GitHub not to upload temporary junk files.
How to set it up (Quick Start)
If you’re using the terminal or the GitHub website, you can start by creating these folders. Here is a handy tip: GitHub won't let you save an empty folder. To get around this, we usually put a tiny, hidden file called .gitkeep inside them.
Kevin’s Pro Tip: Since this is a school project, make your repository Private first! You don't want a classmate accidentally "borrowing" your code and getting you both in trouble for plagiarism. You can always make it public later to show off to recruiters!
Why this works
Think of your notebooks/ folder like a sketchbook. It’s where you try things, fail, and see what happens. Your src/ folder is the final painting. Keeping them separate means that when your professor looks at your code, they see the polished version immediately.
One little thing: If "SAGE" is actually a specific tool you're using (like SageMath), we might want to add a scripts/ folder specifically for .sage files. Does your syllabus mention using "SageMath" or is "SAGE" just the project name?
